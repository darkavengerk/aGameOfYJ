#### 1. 이미지 생성 규칙 (General Image Rules)
* **파일 형식:** 반드시 `.jpg` 또는 `.png` 형식을 사용하며, 웹 호스팅(GitHub)이 가능하도록 파일명에 공백이나 특수문자를 피할 것.
* **해상도 단위:** TTS는 2의 거듭제곱 단위(예: 1024x1024, 2048x2048)에서 가장 잘 작동함. 최대 4096x4096을 넘지 않도록 권장.
* **파일 크기:** 로딩 속도를 위해 장당 2MB 이하로 최적화할 것. 파이썬 `Pillow` 라이브러리를 사용해 압축률을 조절해야 함.
* **색상 프로필:** RGB 모드 사용 (CMYK 사용 시 색상 왜곡 발생 가능).

#### 2. 카드 시트 제작 규칙 (Deck Sheet Rules)
* **그리드 구성:** 가로(Width)와 세로(Height)의 칸수를 명시할 것 (최대 가로 10개 x 세로 7개 = 총 70장).
* **단일 이미지 파일:** 모든 카드는 개별 파일이 아닌, 하나의 큰 '시트(Atlas)' 이미지로 합쳐서 생성할 것.
* **마지막 칸의 비밀:** 시트의 **마지막(70번째 혹은 설정한 장수의 마지막) 칸**은 게임 내에서 카드가 뒤집혀 있거나 보이지 않을 때 표시되는 '숨김 이미지'로 쓰이는 경우가 많음. 따라서 데이터용 카드는 마지막 칸 앞까지만 배치할 것.
* **카드 뒷면:** 모든 카드의 뒷면이 동일하다면 별도의 단일 이미지 파일로 생성하고, 카드마다 뒷면이 다르다면 앞면 시트와 동일한 규격의 '뒷면 시트'를 생성할 것.

#### 3. JSON 파일 생성 및 수정 규칙 (JSON Metadata Rules)
* **구조 유지:** TTS 세이브 파일의 `ObjectStates` 배열 구조를 정확히 지킬 것.
* **CustomDeck ID:** 각 덱은 고유한 `FaceURL`과 `BackURL`을 가지며, `NumWidth`와 `NumHeight` 값이 실제 이미지 시트와 일치해야 함.
* **GUID:** 모든 객체는 6자리의 고유한 `GUID`를 가져야 함. 파이썬으로 생성 시 중복되지 않도록 무작위 문자열 생성 로직을 포함할 것.
* **URL 경로:** `FaceURL`과 `BackURL` 등 모든 이미지 경로는 GitHub의 `raw.githubusercontent.com/...` 형태의 직접 접근 주소로 작성되어야 함.

#### 4. 기타 시스템 규칙 (System & Physics)
* **두께 및 크기(Scale):** 보드판의 경우 `Transform` 내의 `scaleZ` 값을 조절하여 적절한 두께감을 줄 것.
* **스냅 포인트(Snap Points):** 보드판 위에 카드가 놓일 위치가 정해져 있다면, 해당 좌표를 JSON의 `SnapPoints` 리스트에 미리 계산해서 넣을 것.
* **태그(Tags):** 스크립트 활용을 위해 각 카드나 컴포넌트에 `CardType`, `Player1` 등의 태그를 JSON 상에 부여할 것.
